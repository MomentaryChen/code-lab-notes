# Research & Design Decisions — 001-rate-limit-animations

**Feature**: 限流策略可視化教學（固定視窗、滑動視窗、Token Bucket）  
**Branch**: `001-rate-limit-animations`  
**Date**: 2026-03-05  
**Spec**: `specs/001-rate-limit-animations/spec.md`

本檔案彙整本功能在技術選型與設計上的關鍵決策，並依「Decision / Rationale / Alternatives considered」格式記錄。

---

## 1. 實作型態：純前端模擬 vs. 需要後端服務

- **Decision**: 採用純前端模擬（瀏覽器端計算請求事件與限流行為），不新增任何後端 API。  
- **Rationale**:
  - 規格屬教學型互動視覺化，並非線上真實流量治理；不需要真實後端流量或持久化。
  - 專案既有部署目標為 GitHub Pages 靜態網站，純前端模擬可與現有流程完全相容。
  - 純前端有助於使用者在本地或離線（已快取）環境重播相同案例，利於教學與 demo。  
- **Alternatives considered**:
  - 透過後端 API 回傳模擬結果：會增加額外服務與部署複雜度，與憲章中「避免過度設計、以靜態部署為預設」相違，且教學價值有限。
  - 使用第三方 sandbox 或雲服務（如 serverless function）進行模擬：同樣增加外部依賴與治理成本，對本教學場景沒有必要。

---

## 2. 動畫技術：DOM + CSS / React state vs. Canvas / SVG 圖表庫

- **Decision**: 優先採用 DOM + CSS + React state 驅動動畫（以 flex/grid 呈現時間軸與請求事件），暫不引入 Canvas/SVG 圖表庫。  
- **Rationale**:
  - 現有專案既有演算法視覺化皆以 DOM + CSS 為主，延續相同技術堆疊可降低認知成本並保持風格一致。
  - 限流行為的核心在於「時間視窗與請求通過／拒絕狀態」，以圖示／色塊在時間軸上排列即可清楚表達，無須高階圖表能力。
  - DOM + CSS 較易透過 React Testing Library 做互動測試與狀態驗證。  
- **Alternatives considered**:
  - Canvas + 低階繪圖：適合高密度、連續時間序列圖，但實作與重構成本高，且對學習者來說不一定比較直覺。
  - SVG + 圖表庫（例如 chart 套件）：雖可快速產生折線／長條圖，但會增加 bundle 體積與外部 API，需要額外學習與維運成本。

---

## 3. 策略模擬方式：以「請求腳本」驅動 vs. 依實際時間連續模擬

- **Decision**: 採用「請求腳本（request script）」概念：以一組離散的請求時間點陣列作為輸入，將其套用三種策略演算法，輸出事件序列後再驅動動畫。  
- **Rationale**:
  - 可重播性佳：同一組腳本可在三種策略間反覆比較，符合 spec 中「固定流量案例跨策略比較」的需求。
  - 測試友善：在單元測試中能明確給定輸入陣列，驗證演算法是否產出正確的通過／拒絕標記。
  - 動畫與演算法解耦：演算法只關心輸入輸出，動畫僅負責視覺呈現。  
- **Alternatives considered**:
  - 以 `setInterval` 或真實 `Date.now()` 連續模擬：行為較接近真實服務，但不利於精準測試與教學（每次播放結果可能略有差異）。
  - 完全由使用者即時點擊按鈕產生請求：互動感強，但難以用同一組輸入對比三種策略，且不利於編寫 deterministic 測試。

---

## 4. 策略支援範圍：實作哪幾種限流演算法

- **Decision**: 僅實作三種策略：固定視窗（Fixed Window）、滑動視窗（Sliding Window）、Token Bucket，並在 UI 中明確標示名稱與概念說明。  
- **Rationale**:
  - 已完全覆蓋 spec 中明確點名的三種方式，符合需求且不超出範圍。
  - 這三種策略在教學與實務中都相當常見，對理解限流 trade-off 已足夠。
  - 過多策略反而會稀釋學習重點，與 MVP 優先原則不符。  
- **Alternatives considered**:
  - 再加入 Leak Bucket、基於滑動視窗計數的延伸變形：雖可增加完整性，但會使 UI 與文案變複雜，且對當前學習目標非必要。

---

## 5. 參數調整能力：固定 vs. 可調整範圍

- **Decision**: 提供有限且經過保護的參數調整（例如限流速率、視窗大小、桶容量與補充速率），並設置建議範圍與極端值提示，不提供任意輸入。  
- **Rationale**:
  - 給太自由的數值輸入容易產生「幾乎看不到變化」或「畫面極度混亂」的情況，會降低教學效果。
  - 建議範圍有助於保持動畫在可理解的密度與長度之內，符合「高品質教學體驗」目標。
  - 以極端值作為特例（例如非常小的配額或非常大的桶）反而可以強化對概念的理解。  
- **Alternatives considered**:
  - 允許完全自由的參數輸入：需要更多錯誤處理與 UI 防護，且對學習幫助有限。
  - 僅提供固定預設組合（無滑桿或輸入框）：雖然簡單，但不利於體驗「參數 → 行為變化」的動態關係。

---

## 6. UI 整合位置：獨立新頁 vs. 併入 SystemArchitecturePage

- **Decision**: 將限流可視化區塊作為 `SystemArchitecturePage` 中「限流」小節的延伸內容，而非獨立新路由。  
- **Rationale**:
  - 現有 changelog 顯示系統架構頁已收納「限流」與相關概念，本功能屬於該主題的深化教學，自然應位於同一頁面。
  - 不新增多餘的 route 與導航項目，可減少使用者在「閱讀說明」與「觀看動畫」之間切換成本。
  - 符合憲章中對「避免過度設計」與簡潔導覽的要求。  
- **Alternatives considered**:
  - 新增獨立 `/rate-limiting` 或 `/system-architecture/rate-limiting` 頁面：較有彈性，但會增加路由與導覽複雜度，目前教學範圍尚不需要。

---

## 7. 測試策略：演算法 vs. UI 動畫

- **Decision**: 以「演算法核心邏輯單元測試 + 關鍵互動行為測試」為主，不追求逐幀驗證 CSS 動畫過程。  
- **Rationale**:
  - 演算法輸出（對每個請求事件的通過／拒絕標記）才是教學正確性的關鍵，這部分應以單元測試嚴格覆蓋。
  - UI 動畫細節易受 CSS 或瀏覽器差異影響，過度追蹤會讓測試脆弱且維護成本高。
  - 以互動測試檢查「策略切換、參數調整後，畫面上事件狀態改變是否符合演算法輸出」即可充分保證行為。  
- **Alternatives considered**:
  - 以快照測試（snapshot）完整比對動畫 DOM 結構：對視覺化類 UI 來說過度敏感，容易因微調樣式頻繁失敗。
  - 使用錄影或 timeline 工具做自動視覺驗證：部署與維護成本過高，不符合本專案教學型質。
