# Feature Specification: 限流策略可視化教學（固定視窗、滑動視窗、Token Bucket）

**Feature Branch**: `001-rate-limit-animations`  
**Created**: 2026-03-05  
**Status**: Draft  
**Input**: User description: "限流中 加入各種實作方式  固定視窗、滑動視窗、Token Bucket等  每一種方式都要有動畫可以讓人更快理解作法"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 以動畫理解單一限流策略 (Priority: P1)

一位剛接觸「限流」概念的學習者打開教學頁面，選擇固定視窗限流，透過播放動畫看到請求在時間軸上的分布、哪些被通過、哪些被拒絕，並搭配文字說明理解「在一段固定時間內最多只能通過幾個請求」的核心概念。

**Why this priority**: 這是多數使用者的第一個接觸點，也是整個限流教學的基礎；只要單一策略的行為能被看懂，就已經具備實用的入門收穫，是最小可行教學單元（MVP）。

**Independent Test**: 只開放固定視窗限流的動畫與說明，請測試者觀看 3 分鐘後回答幾個簡單情境題（例如「在這段時間再發出一個請求會被怎麼處理？」），若多數人能正確回答且能自己描述固定視窗限流的原理，則此故事可被獨立驗證通過。

**Acceptance Scenarios**:

1. **Given** 使用者首次進入限流教學頁面，**When** 選擇固定視窗並按下播放或開始按鈕，**Then** 會看到時間軸上請求一個個出現並依照固定視窗規則標示為「通過」或「被拒絕」。
2. **Given** 動畫播放進行中，**When** 使用者暫停並把滑鼠或游標移到某一個請求事件上，**Then** 會看到該事件所屬時間視窗、目前視窗內已通過的請求數、以及為何被通過或被拒的文字說明。

---

### User Story 2 - 比較不同限流策略的行為差異 (Priority: P2)

一位具備一定背景的工程師或學生希望理解固定視窗、滑動視窗與 Token Bucket 在同樣流量模式下的差異，於是透過切換策略按鈕或分頁，將同一組請求流量套用在三種策略上觀察動畫，包含瞬間流量尖峰時被拒比例、請求通過的平滑程度等。

**Why this priority**: 不同策略的優缺點與適用情境是限流設計很重要的決策點，透過對比式動畫可以快速看出差異，幫助學習者在實務上做出更合理的選擇。

**Independent Test**: 僅提供一組固定的請求流量腳本與三種策略切換，請測試者依序觀看三種策略的動畫後，回答「哪一種策略在尖峰時比較容易拒絕請求」、「哪一種策略較能接受短暫突發但長期受控」等比較題，若多數人能根據動畫正確回答，則此故事成立。

**Acceptance Scenarios**:

1. **Given** 使用者選擇一個預設的「尖峰流量」案例，**When** 在三種策略之間切換，**Then** 每種策略都會以相同的請求時間分布播放動畫，並在畫面上清楚區分各自的通過與拒絕情況。
2. **Given** 三種策略都有對應的文字摘要或標籤，**When** 使用者觀看動畫結束或暫停時，**Then** 可在旁邊看到該策略的高層次說明（例如是否允許短暫突發、是否較平滑、適合的典型場景）。

---

### User Story 3 - 透過參數調整體驗限流策略的行為變化 (Priority: P3)

一位進階學習者希望體驗不同「每秒允許次數」、「視窗大小」、「桶容量」、「補充速率」等參數對限流效果的影響，因此在教學頁面的控制列上調整參數後，重新播放動畫，觀察請求通過比例、排隊或被拒情況的變化。

**Why this priority**: 參數調整是實際設計限流時必不可少的步驟，透過互動式操作可以幫助學習者建立對「參數大小」與「實際效果」之間關係的直覺。

**Independent Test**: 僅提供一個策略（例如 Token Bucket）與一組可調的參數，請測試者分別使用「較小的桶容量」與「較大的桶容量」播放動畫，若多數人能觀察到可接受請求的突發程度不同並能口頭描述差異，則此故事可以獨立驗證。

**Acceptance Scenarios**:

1. **Given** 使用者開啟參數設定面板，**When** 調整限流速率或桶容量等數值並重新播放動畫，**Then** 畫面上請求通過與被拒的節奏會明顯隨參數改變而改變，且不需要重新載入整個頁面。
2. **Given** 使用者將參數調整到極端值（例如非常小的允許次數或非常大的桶容量），**When** 再次播放動畫，**Then** 可以明顯看出「幾乎全部被拒」或「短時間內幾乎全部通過」等極端行為，並有文字提示提醒這是極端設定。

---

### Edge Cases

- 在流量遠低於限流上限時，所有策略應表現為「幾乎所有請求都通過」，動畫不應出現不合理的拒絕標記。
- 在極端尖峰流量（短時間內大量請求）情境下，各策略的動畫應仍然可讀，不會因過多標記而難以分辨，並有方式顯示總數統計。
- 當使用者快速重複播放、暫停或切換策略時，動畫狀態不應卡死或出現殘留標記，畫面能回到一致且容易理解的狀態。
- 若使用者在動畫進行中調整參數，系統需明確決定是「立即以新參數重新演算動畫」或「提示需重新播放」，避免一部分畫面用舊參數、一部分用新參數導致誤解。
- 當頁面停留很久或瀏覽器被最小化後再回來時，動畫狀態與時間軸應有合理的重設或暫停邏輯，以避免使用者誤以為系統持續在背景運算。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 教學頁面 MUST 提供一個清晰的導言區塊，以非技術語言說明為何需要限流，以及固定視窗、滑動視窗、Token Bucket 這三種常見策略的大致用途與差異。
- **FR-002**: 系統 MUST 為每一種限流策略提供獨立且可重複播放的動畫展示區，使用者可以選擇其中一種策略並看到請求在時間上的流動與被通過／被拒絕的結果。
- **FR-003**: 系統 MUST 在動畫附近提供關鍵狀態的即時標示，例如目前時間區間內已通過的請求數、剩餘可用額度、桶中剩餘資源數量等，讓使用者可以對照動畫與文字理解當前狀態。
- **FR-004**: 系統 MUST 提供簡單易用的控制列，至少包含開始／暫停播放、重播、調整播放速度，以及在預設案例與自訂參數之間切換的能力。
- **FR-005**: 系統 MUST 提供至少一組預設案例（例如「平穩流量」、「尖峰流量」、「週期性突發」），讓使用者可以在三種策略之間以相同流量型態做對比。
- **FR-006**: 系統 MUST 提供一個對比說明區塊，以表格或條列方式整理三種策略在「是否允許突發」、「是否平滑」、「典型適用場景」等維度上的差異，並與動畫行為保持一致。
- **FR-007**: 系統 MUST 支援使用者在不重新整理頁面的情況下切換策略或案例，並確保畫面傳達的限流行為與當前選擇相符，避免殘留任何前一種策略的狀態解說。
- **FR-008**: 系統 MUST 以明確易懂的錯誤或提示訊息回應不合理的參數設定（例如明顯超出建議範圍），避免讓使用者產生「動畫壞掉」的誤會。

### Key Entities *(include if feature involves data)*

- **限流策略（Rate Limiting Strategy）**: 代表固定視窗、滑動視窗、Token Bucket 等不同概念單元，包含名稱、文字說明、是否允許短期突發、是否平滑、適用情境等屬性。
- **請求事件（Request Event）**: 代表在時間軸上的單一次請求，包含發生時間、是否被通過或拒絕、依據的限流規則說明等屬性，用於驅動動畫展示。
- **流量案例（Traffic Scenario）**: 代表一組預先設計或使用者自訂的流量模式，如平穩、尖峰、週期性突發，包含請求分佈與描述文字，以利在不同策略之間重複使用。
- **參數組態（Rate Limit Parameters）**: 代表各策略對應的可調整參數，例如單位時間允許請求數、時間視窗長度、桶容量、資源補充速度等，用於產生對應行為。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在可用性測試中，至少 80% 的初學者在觀看固定視窗限流動畫與說明不超過 5 分鐘後，能用自己的話正確描述其基本原理與限制。
- **SC-002**: 在比較三種策略的情境下，至少 80% 的受測者能根據動畫行為正確回答哪一種策略較適合「平滑長期流量」、哪一種較適合「允許短暫突發但長期受控」。
- **SC-003**: 至少 80% 的受測者在透過調整參數並觀看動畫後，能清楚說出「把參數調大或調小」會對被通過與被拒請求的分佈造成什麼方向性的影響。
- **SC-004**: 教學頁面發佈後，針對限流概念的相關求助或疑問（例如在課程、文件或社群中提出）數量，能在後續一段追蹤期間內明顯下降，或使用者回饋顯示對限流策略的理解程度明顯提升。
